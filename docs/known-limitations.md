# TENTENS — 알려진 한계 & 기술부채 상태

> 기술부채 정리 세션 기록. 해결/보류 상태를 한눈에 본다.

## 해결됨

### ✅ 멀티 모드 점수 서버 재계산 (anti-cheat)
- **문제였던 것**: 소켓 `game:result` 가 클라이언트가 보낸 점수를 그대로 믿고 방 순위를 매겼음 → 방 내 승자 조작 가능.
- **해결**: `server/scoring.ts` 가 프론트 evaluator(`@/lib/poker/evaluator`)를 재사용해 보드(slots)로 서버에서 재계산. `game:result` 는 이제 `slots` 만 받아 서버 권위 점수/타이브레이커/조합을 산출한다. 클라이언트는 점수를 전송하지 않는다.
- **참고**: 글로벌 리더보드(`/api/leaderboard`)는 싱글·멀티 모두 이미 서버 재계산이었다. 이번 변경으로 방 순위 표시까지 서버 권위값으로 일관됨.

### ✅ evaluator 서버 공용화
- tsx 가 `tsconfig` 의 `@/*` alias 를 런타임 해석하므로 커스텀 서버가 프론트 evaluator 를 그대로 import 한다. 판정 로직 중복 없음(단일 소스).

### ✅ README 문서-코드 불일치
- `server.ts` 단일 파일로 오기재된 것을 실제 `server/` 디렉토리 구조로 정정. 실행 명령(`yarn`, `tsx server/index.ts`) 명확화.

## 보류 (출시 직전 재검토)

### ⏸ 진행 중 게임 상태 인메모리 (#3)
- **현황**: 방/게임 상태를 `server/state.ts` 의 인메모리 `Map` 에 보관. 서버 재시작 시 진행 중 게임 소실.
- **결정**: **지금은 보류.** 소규모 + Railway 단일 인스턴스라 재시작 빈도가 낮고, Redis 도입은 인프라 비용/복잡도를 늘린다. 동접이 늘거나 다중 인스턴스로 확장할 때 Redis(또는 Upstash) 도입 검토.

### ⏸ Supabase RLS anon 전면 개방 (#4)
- **현황**: 앱이 anon 키 기반이라 모든 테이블에 anon 전면 rw 정책. 마이그레이션 주석에도 "출시/광고 직전 service_role 전환 시 강화"로 명시됨.
- **결정**: **출시 직전으로 보류.** 지금 service_role 로 전환하면 로그인/리더보드/친구 기능이 깨질 위험이 커, docs 계획대로 출시 직전 일괄 전환이 안전.

## 후속(더 강한 anti-cheat, 후순위)
- 현재 서버 재계산은 "보드 기반 점수 재산출"까지다. 클라가 보낸 카드가 실제 배분된 덱의 카드인지(덱 위조 방지)는 검증하지 않는다.
- 서버는 `room.deck` 를 이미 보유하므로, 제출된 slots 의 카드가 전부 해당 덱에 속하고 중복이 없는지 검증하는 강화가 가능하다. 필요 시 `server/scoring.ts` 에 덱 대조 단계를 추가.
