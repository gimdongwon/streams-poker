# TENTENS 앱화 로드맵 (설계)

> 작성일: 2026-07-02
> 상태: 승인됨 (브레인스토밍 산출물 — 하위 프로젝트 분해/로드맵)
> 관련 문서: [streams-monetization-plan.md](../../streams-monetization-plan.md), [known-limitations.md](../../known-limitations.md)

## 목표

Next.js 웹 게임 TENTENS를 네이티브 앱으로 출시한다. 이 문서는 단일 구현 스펙이 아니라
**여러 독립 하위 프로젝트로의 분해와 실행 로드맵**이다. 각 하위 프로젝트는 별도의
스펙 → 계획 → 구현 사이클을 갖는다.

## 확정된 핵심 결정

1. **래핑 방식: A — 원격 URL 웹뷰 래퍼.**
   - Capacitor 웹뷰가 호스팅된 URL을 로드한다. 네이티브가 그리는 화면은 없다(대문 포함).
   - 모든 페이지(로비/대기방/게임/결과)는 지금 그대로 웹뷰 안에서 돈다.
   - 네이티브는 "페이지"가 아니라 **기기 기능**만 담당: AdMob 광고, 스플래시, 상태바, 뒤로가기 등.
   - 근거: 현재 구조가 커스텀 서버 + API 라우트 + Socket.io(동일 origin)라 `next export`가 불가능하다.
     프론트를 정적 번들(방식 B)하려면 프론트/백엔드 분리 대공사가 필요해 YAGNI에 어긋난다.
     실시간 멀티가 본질이라 오프라인 이점도 없다. 방식 A가 monetization 문서의
     "웹뷰=게임 / 네이티브=광고" 역할 분리와 정확히 일치한다.

2. **플랫폼: 안드로이드 우선, iOS는 후속 트랙.**
   - iOS도 같은 Capacitor 코드베이스에서 `cap add ios`로 가능하나, Mac+Xcode, Apple Developer
     $99/년, 그리고 **가이드라인 4.2(최소 기능)** 로 순수 웹뷰 래퍼가 리젝될 리스크가 있어
     비용·심사 리스크가 낮은 안드로이드를 먼저 완주한다.
   - iOS는 ②의 산출물을 재사용하고 ③/④ 이후 별도 심사 대응 트랙으로 둔다.

## 의존성 그래프

```
② Capacitor 래핑 ──→ ③ 스토어 출시 ──→ ④ AdMob 보상형 광고
                           ↑
   인프라: Railway 유지 (안정 HTTPS/WSS·app-ads.txt 이미 충족)

[후속 트랙] iOS 래핑 — ②의 산출물 재사용, ③/④ 이후 별도 심사
```

인프라는 Railway(`www.tentens.kr`)를 유지한다. Fly.io 이전은 평가 후 보류했다(아래 ① 참고).

## 하위 프로젝트

### ① 인프라 — Railway 유지 (Fly.io 이전 보류)
- **결정(2026-07-02)**: Fly.io 이전은 **하지 않는다.** 현재 Railway 유료 티어에서 안정 운영 중이고
  도메인 `www.tentens.kr` 도 연결돼 있다. Fly 이전 동기(비용/안정성)가 지금은 유효하지 않다
  (Fly도 무료 티어가 사실상 없어져 비용 이점 불확실).
- **현황**: 스토어 출시(③)·광고(④)에 필요한 인프라 요건(안정 HTTPS/WSS 엔드포인트, `app-ads.txt` 호스팅)을
  Railway가 이미 충족한다.
- **제약**: 인메모리 방 상태(기술부채 #3) 때문에 **단일 인스턴스로 유지**(레플리카 금지). Redis 도입 전까지 하드 제약.
- **재검토 조건**: Railway 비용 부담 또는 실제 장애를 겪을 때. (Fly 아티팩트는 git 히스토리에 있어 필요 시 복구 가능)

### ② Capacitor 래핑 (안드로이드)
- **목표**: 설치되는 AAB/APK, 웹뷰가 원격 URL 로드, 기본 네이티브 기능.
- **핵심 작업**: `@capacitor/core`+`@capacitor/android`, `capacitor.config`(server.url=호스팅 도메인),
  앱 아이콘/스플래시, 상태바·안드로이드 뒤로가기 처리, safe-area, 화면 방향(게임 가로 우선 — `OrientationGate` 존재).
- **완료 기준**: 실기기 설치 → 로비~게임~결과 정상, 뒤로가기/스플래시 정상.
- **참고**: `server.url` 은 이미 운영 도메인 `https://www.tentens.kr`(Railway) 를 가리킨다.

### ③ 구글플레이 출시
- **목표**: 최소 내부 테스트 트랙 → 프로덕션 게시.
- **핵심 작업**: 서명 키스토어, AAB 빌드, 스토어 등록정보(아이콘/스크린샷/설명), 개인정보처리방침 URL(이미 `/privacy` 존재),
  데이터 보안 양식, 연령/콘텐츠 등급, "poker" 키워드 회피.
- **완료 기준**: 플레이스토어 게시(내부 테스트 이상).

### ④ AdMob 보상형 광고 ("1.5배 점수")
- **목표**: 자발적 광고 시청 → 이번 판 점수 1.5배 (Ready 강제 아님, 정책 준수).
- **핵심 작업**: `@capacitor-community/admob`, 보상형 광고 로드/표시, JS 브리지 연동, 1.5배의 서버 검증.
- **결정 포인트**: 광고 시청은 클라 이벤트라 위조 가능 → **AdMob 서버측 검증(SSV)** 로 서버가 1.5배를 신뢰.
  이미 구현된 서버 권위 점수 로직과 연결된다.
- **완료 기준**: 광고 시청 후 1.5배가 리더보드에 반영, AdMob 정책 통과.
- **선행조건**: 스토어 정식 출시(③)가 있어야 AdMob 심사 가능.

## 권장 실행 순서

1. ② 래핑 (완료 — 에뮬레이터 동작 확인). 프론트 배포 시 네이티브 동작 활성화.
2. ③ 스토어 출시 — 내부 테스트 → 프로덕션 (+ #4 RLS 강화). 인프라는 Railway 유지.
3. ④ AdMob — 출시·심사 통과 후.
4. [후속] iOS 트랙 — 같은 래핑 재사용 + 4.2 심사 대응.

## 보류했던 기술부채와의 연결

- **#3 인메모리 방 상태**: Railway를 단일 인스턴스로 유지(레플리카 금지)하는 제약으로 재등장. 동접이 커지면 Redis 도입 필요.
- **#4 anon RLS 개방**: ③ 스토어 출시 직전 service_role 기반으로 강화하는 것이 이 시점의 과제.
- **서버 권위 점수(이미 구현)**: ④의 "1.5배 치팅 방지"의 토대.
